1. PROBLEMAS CRITICOS (SEGURIDAD Y DATOS)
================================================================================


[CRITICO] 1.1 SECRET_KEY EXPUESTA EN CODIGO FUENTE
  Archivo: nuxt.config.ts, linea 17
  Problema: La llave de encriptacion 'THESALUS943875PL' esta
            hardcodeada en el codigo fuente.
  Riesgo:   Cualquiera con acceso al repositorio puede ver la llave
            y descifrar los tokens y datos de usuarios.
  Solucion: Mover a archivo .env y usar useRuntimeConfig().public.SECRET_KEY
            sin exponer el valor en el codigo.


[CRITICO] 1.2 TOKEN EN LOCALSTORAGE
  Archivo: Core/Login/Ingresar.js
  Problema: El token de autenticacion se guarda en localStorage
            encriptado con crypto-js, pero la llave esta en el
            mismo frontend. Un atacante podria descifrarlo.
  Solucion: Considerar usar httpOnly cookies para el token, o
            un mecanismo de refresh token con backend.


[CRITICO] 1.3 API URL DE DESARROLLO EN PRODUCCION
  Archivo: nuxt.config.ts, linea 19
  Problema: La URL activa es 'http://127.0.0.1:8000' (local).
            La URL de produccion esta comentada.
  Riesgo:   Si se genera un build sin verificar, la app apuntara
            al servidor local en vez del produccion.
  Solucion: Usar variables de entorno (.env) para URL de API
            y descomentar la URL de produccion.


[ALTO] 1.4 LIMPIEZA DE LOCALSTORAGE INCOMPLETA
  Archivo: pages/index.vue, lineas 44-48
  Problema: Se limpian token, user, Rol, Permisos y
            permisosTemporales, pero no se limpia 'seccion',
            'Empresa' ni 'ultimaSincronizacion'.
  Solucion: Consolidar la limpieza en una funcion centralizada
            y asegurar que todos los datos se eliminen al cerrar sesion.

2. PROBLEMAS DE CODIGO (BUGS Y ERRORES)
================================================================================


[ALTO] 2.1 VARIABLE NO DECLARADA EN Vadecum.vue
  Archivo: pages/Empresas/Codigos/Vadecum.vue, linea 144
  Problema: Se usa 'acciones.length' pero la variable se llama
            'accionesVademecum'. Esto generara un error de
            referencia o funcionara por coincidencia con otra variable.
  Codigo actual:
    if (acciones.length > 0) {  // <-- Deberia ser accionesVademecum
    builderTablaVademecum.setAcciones({ icons: accionesVademecum, botones: true });
  Solucion: Cambiar 'acciones.length' por 'accionesVademecum.length'.


[ALTO] 2.2 VARIABLE 'convenios' NO DEFINIDA EN Usuarios/index.vue
  Archivo: pages/Usuarios/index.vue, linea 380
  Problema: Se pasa ':historias="convenios"' al componente CardKardex,
            pero 'convenios' no esta definida como ref() en el
            script setup. Se declara como variable local en
            llamaConvenios() pero no como ref reactiva.
  Solucion: Declarar 'const convenios = ref([])' y popularlo
            desde llamaConvenios().


[MEDIO] 2.3 CONSOLE.LOG EN PRODUCCION
  Archivos: multiples archivos en pages/, Core/, composables/
  Problema: Hay multiples console.log() que deberian eliminarse
            o reemplazarse con un sistema de logging apropiado.
  Ejemplos:
    - pages/index.vue: linea 34, 37
    - pages/Historial/index.vue: linea 271, 383
    - Core/Login/Ingresar.js: linea 82
  Solucion: Eliminar console.log() o usar un logger configurable
            que se pueda desactivar en produccion.


[MEDIO] 2.4 FUNCION vacia validarCodigo()
  Archivo: pages/index.vue, linea 98-100
  Problema: La funcion validarCodigo() esta declarada vacia pero
            se pasa como prop al builder de recuperar contrasena.
  Solucion: Implementar la logica o eliminar la referencia.


[MEDIO] 2.5 CODIGO COMENTADO EXTENSO
  Archivo: pages/Historial/index.vue, lineas 302-336
  Problema: Hay un bloque grande de codigo comentado con un switch
            case que parece ser la implementacion anterior de
            carga de datos. Esto genera confusion.
  Solucion: Eliminar el codigo comentado. Usar git para mantener
            el historial de cambios.


[BAJO] 2.6 DUPLICACION DE IMPORTACIONES
  Archivo: pages/Usuarios/Profesional/index.vue, linea 22
  Problema: 'useProfesionalStore' se importa dos veces (lineas 15
            y 22) desde el mismo archivo.
  Solucion: Eliminar la importacion duplicada.


[BAJO] 2.7 PUNTO Y COMA DUPLICADO
  Archivo: pages/Home.vue, linea 10
  Problema: Doble punto y coma en el import:
            import { useProfesionalStore } from '~/stores/Entidades/Profesional';;

[BAJO] 2.8 TYPO EN COMENTARIO
  Archivo: pages/Usuarios/index.vue, linea 31
  Problema: Comentario dice "cunso" en vez de "cuando":
            // Actualizar pagina cunso se agrega Nuevo Usuario

[BAJO] 2.9 TYPO EN VARIABLE

3. MEJORAS EN LA EXPERIENCIA DE USUARIO (UX)
================================================================================


[ALTO] 3.1 NAVEGACION CON location.href
  Archivos:
    - pages/Home.vue, linea 397: location.href = '/Historial'
  Problema: Se usa location.href en vez del router de Vue, lo cual
            provoca una recarga completa de la aplicacion y pierde
            el estado global.
  Solucion: Usar router.push('/Historial') o useRouter().push().


[MEDIO] 3.2 FALTA CONFIRMACION ANTES DE CERRAR SESION
  Archivo: data/NavMenu.js
  Problema: Al cerrar sesion, se eliminan datos directamente sin
            preguntar al usuario si esta seguro.
  Solucion: Agregar una alerta de confirmacion antes de cerrar sesion
            (especialmente si hay datos sin sincronizar).


[MEDIO] 3.3 FALTA INDICADOR DE CARGA CONSISTENTE
  Problema: El indicador de carga (varView.cargando) se activa en
            algunos lugares pero no en todos los procesos largos.
            Algunas operaciones no muestran feedback al usuario.
  Solucion: Asegurar que toda operacion asincrona muestre el
            indicador de carga o un toast de progreso.


[MEDIO] 3.4 MODALES QUE NO SE CIERRAN AL ENVIAR
  Problema: En algunos formulares, despues de enviar los datos
            correctamente, el usuario debe cerrar manualmente
            el modal.
  Solucion: Cerrar el modal automaticamente despues de una
            operacion exitosa y mostrar un toast de confirmacion.


[BAJO] 3.5 MENSAJES DE ERROR GENERICOS
  Archivo: Core/Login/Ingresar.js, linea 76
  Problema: El mensaje de error es "Informacion invalida" sin
            especificar que fallo (correo o contrasena).
  Solucion: Mostrar mensajes mas especificos cuando sea seguro
            hacerlo (sin comprometer seguridad).


[BAJO] 3.6 FALTA VALIDACION EN TIEMPO REAL DE FORMULARIOS
  Problema: Algunos formularios solo validan al enviar, lo que
            puede frustrar al usuario si tiene muchos campos
            llenos con un error al final.
  Solucion: Agregar validacion en tiempo real (onBlur) para
            campos criticos como documento, correo, fecha.

4. MEJORAS EN LA ARQUITECTURA
================================================================================


[ALTO] 4.1 CENTRALIZAR LA CARGA DE DATOS INICIALES
  Archivo: Core/BDload.js
  Problema: La carga de datos al login es monolitica. Si un
            endpoint falla, puede afectar toda la carga.
  Solucion: Implementar carga independiente por modulo, con
            reintentos y manejo de errores individualizado.


[MEDIO] 4.2 SEPARAR LOGICA DE PERMISOS
  Problema: Los permisos se verifican en multiples archivos
            con la misma logica: varView.getPermisos.includes('X').
  Solucion: Crear un composable usePermisos() que centralice
            todas las verificaciones:
              const { puedeVerPacientes, puedeCrearPacientes } = usePermisos()


[MEDIO] 4.3 REDUCIR DUPLICACION EN WATCHERS
  Problema: Hay multiples watchers con la misma estructura
            (verificar cambioEnApi, llamar datos, incrementar refresh).
            Este patron se repite en casi todos los modulos.
  Solucion: Crear un composable useActualizacionAutomatica() que
            maneje este patron comun:
              useActualizacionAutomatica({
                  store, llamadatos, refresh, showRef
              })


[MEDIO] 4.4 CONSOLIDAR CONSTRUCCION DE TABLAS
  Problema: Las columnas de tablas se definen inline en muchos
            archivos, lo que dificulta mantener consistencia.
  Solucion: Crear un directorio data/columns/ con las definiciones
            de columnas reutilizables.


[MEDIO] 4.5 NOMBRE INCONSISTENTE DE DIRECTORIO
  Problema: El directorio de layouts se llama 'Layouts/' con L
            mayuscula, pero Nuxt espera 'layouts/' en minusculas
            para el file-based routing automatico.
  Solucion: Verificar que Nuxt reconoce el directorio. Si no,
            renombrar a 'layouts/'.


[BAJO] 4.6 FALTA TIPO DE DATOS EN FORMULARIOS
  Problema: Los stores de formularios no tienen definicion de
            tipo para los campos (es JavaScript puro).
  Solucion: Considerar migrar a TypeScript para mejor
            autocompletado y deteccion de errores en tiempo de
            compilacion.


[BAJO] 4.7 COMENTARIOS CON EMOJI EN CODIGO
  Problema: Hay comentarios con emoji (✅, ⚠️, ❌, 📊, 📋, etc.)
            que pueden no renderizarse correctamente en todos
            los editores y terminales.
  Solucion: Reemplazar por texto plano o usar una convencion
            de comentarios estandar.

6. RENDIMIENTO
================================================================================


[ALTO] 6.1 CARGA MASIVA DE DATOS AL LOGIN
  Archivo: Core/BDload.js
  Problema: Al iniciar sesion se descargan todos los codigos
            (CIE10, CUPS, CIF, Vadecum, etc.) de golpe. Si la
            conexion es lenta, esto puede tardar varios segundos.
  Solucion: Implementar carga lazy (bajo demanda) de codigos
            que no se usan inmediatamente. Cargar solo lo esencial
            al login y el resto cuando se necesite.


[MEDIO] 6.2 MULTIPLICACION DE LLAMADAS A LA API
  Archivo: pages/Usuarios/index.vue, onMounted
  Problema: Se hacen 8 llamadas secuenciales a la API al cargar
            el modulo de pacientes (traer pacientes, profesionales,
            insumos, EPS, convenios, etc.). Esto puede ser lento.
  Solucion: Usar Promise.all() para llamadas independientes:
              await Promise.all([
                  pacientesStore.traer(false),
                  llamadatos(),
                  llamaProfesionales(),
                  ...
              ])


[MEDIO] 6.3 RENDERIZADO CONDICIONAL DE FORMULARIOS
  Archivo: pages/Usuarios/index.vue, lineas 333-335
  Problema: Los formularios se renderizan con v-if pero sin
            guardar el estado del formulario cuando se cierra.
            Cada vez que se abre, se reconstruye completo.
  Solucion: Usar v-show o mantener el componente montado con
            un flag de visibilidad para preservar el estado.


[MEDIO] 6.4 FALTA MEMOIZACION EN ALGUNOS COMPUTED
  Problema: Algunos computed se recalculan innecesariamente
            porque dependen de datos que no cambian frecuentemente.
  Solucion: Revisar las dependencias de cada computed y usar
            watch con immediate: false cuando sea necesario.


[BAJO] 6.5 ARCHIVOS GRANDES
  Archivo: pages/Historial/index.vue (923 lineas)
  Problema: Este archivo es muy grande y dificil de mantener.
  Solucion: Extraer logica a composables y componentes separados.
            Un archivo no deberia superar las 300-400 lineas.

7. MANTENIBILIDAD DEL CODIGO
================================================================================


[ALTO] 7.1 CONVENCION DE NOMBRES INCONSISTENTE
  Problema: Hay mezcla de convenciones:
    - Archivos: mayusculas (Home.vue) y minusculas (aprobar-permiso.vue)
    - Directorios: mezcla (Empresas/ y Facturacion/ con espacio)
    - Funciones: camelCase y some_snake_case
    - Variables: mezcla de espanol e ingles (cargando, refresh, show)
  Solucion: Definir y documentar una convencion de nombres:
    - Archivos Vue: PascalCase (Home.vue, NuevoPaciente.vue)
    - Directorios: kebab-case o camelCase
    - Funciones: camelCase en espanol
    - Variables: camelCase en espanol


[MEDIO] 7.2 FALTA DOCUMENTACION DE COMPONENTES
  Problema: Los componentes no tienen JSDoc ni comentarios que
            expliquen sus props, eventos y uso.
  Solucion: Agregar JSDoc a los componentes principales:
              /**
               * TablaNuxt - Componente generico de tabla
               * @prop {Object} propiedades - Configuracion de la tabla
               * @prop {Array} propiedades.columns - Columnas a mostrar
               * @prop {Array} propiedades.data - Datos a renderizar
               * @prop {Function} propiedades.llamadatos - Refrescar datos
               */


[MEDIO] 7.3 ARCHIVO .ENV EXPUESTO
  Archivo: .env esta en el repositorio
  Problema: El archivo .env con posibles secretos esta versionado.
  Solucion: Agregar .env a .gitignore y crear un .env.example
            con las variables necesarias sin valores sensibles.


[MEDIO] 7.4 FALTA TESTS
  Problema: No se encontraron pruebas unitarias ni de integracion.
  Solucion: Implementar al menos:
    - Tests de composables (validaciones, formateo)
    - Tests de stores (acciones CRUD basicas)
    - Tests de Core (funciones de login, envio de datos)
    Framework sugerido: Vitest (compatible con Nuxt 3)


[BAJO] 7.5 CARPETA .output EN EL REPOSITORIO
  Problema: La carpeta .output/ (build generada) esta en el
            repositorio, lo cual增加了 el tamano innecesariamente.
  Solucion: Agregar .output/ y .output.zip a .gitignore.


[BAJO] 7.6 SIN ESLINT CONFIGURADO
  Problema: No se encontro configuracion de ESLint en el proyecto.
  Solucion: Configurar ESLint con las reglas de Nuxt 3 para
            mantener consistencia en el codigo:
              npx nuxi prepare
              npm install -D @nuxt/eslint-config
            Y agregar al package.json el script de lint.
